# zinx-threads

Slack-style team chat. **electron-vite (Vite + TypeScript)** desktop shell; **React 19** UI + **Convex** backend (**full data model live — chat/threads/pages/kanban/voice/shared-channels all real; the app is Convex-only, the mock/demo scaffolding was removed 2026-07-12**, see memory `keep-mock-fallback`) + **WorkOS AuthKit** auth (wired — Path A auto-provision). Electron 39. **pnpm, not npm.**

**Product name = "Zinx Threads"** (display everywhere: `<title>`, sign-in, onboarding, settings, electron `productName`/`appId` `com.zinxthreads.app`). Logo = `components/app-shell/logo.tsx` (`Logo` = `bg-primary` squircle + `text-primary-foreground` "Z", so it follows the theme; `LogoWordmark` = mark + name). Web favicon `src/renderer/public/favicon.svg` (copied to both `out/web` + `out/renderer` builds; `<link rel=icon>` + `theme-color` in `index.html`). **Packaged desktop icon** is generated from the same favicon geometry (rose squircle + white "Z") by `scripts/generate-icon.mjs` — a pure-Node SDF/PNG encoder (no native `sharp`/`resvg` dep) that writes a 1024×1024 PNG to **both** `resources/icon.png` (Linux window + runtime) **and** `build/icon.png` (electron-builder's source; it regenerates `icon.ico`/`icon.icns` from this at package time — the scaffold `.ico`/`.icns` were deleted so the branded PNG is the single source). Re-run `node scripts/generate-icon.mjs` if the logo changes.

**One codebase → desktop AND web.** The `src/renderer` app is the product; the Electron main/preload are just the desktop shell. It also builds as a plain web app (`vite.web.config.ts`) because data goes straight to Convex (works anywhere), native calls go through `src/renderer/src/lib/platform.ts` (feature-detect + web fallback — renderer code must NOT touch `window.electron`/`window.api` directly), and `router.ts` picks history per target (`isPackagedDesktop` → hash for `file://`; web AND `pnpm dev` → browser history). Keep the renderer web-portable: no Electron/Node imports in `src/renderer`, always go through the `platform` layer.

## Commands

- `pnpm dev` — desktop app with HMR (renderer) + hot-reload (main/preload)
- `pnpm dev:web` — the same renderer as a **web app** (Vite, http://localhost:5173)
- `pnpm build` — typecheck, then `electron-vite build` → `out/` (desktop)
- `pnpm build:web` — typecheck:web, then `vite build` → `out/web` (web) · `pnpm preview:web`
- `pnpm start` — preview a production desktop build (`electron-vite preview`)
- `pnpm lint` — ESLint · `pnpm typecheck` — tsc (node + web) · `pnpm format` — Prettier · `pnpm test` — vitest (renderer logic + `convex-test` backend; `test:watch` to watch)
- `pnpm build:win` — build + package installer via electron-builder (also `build:mac`, `build:linux`)

## Process model

Three isolated processes, one `electron.vite.config.ts` with a `main` / `preload` / `renderer` section each.

- `src/main/index.ts` — main process (Node): windows, app lifecycle, OS APIs. Uses `@electron-toolkit/utils`.
- `src/preload/index.ts` — bridge: expose a _narrow, typed_ API via `contextBridge.exposeInMainWorld`. No raw `ipcRenderer`, no Node, to the renderer. Types in `src/preload/index.d.ts` (augments `window.electron` / `window.api`).
- `src/renderer/src/` — renderer (browser): React UI only. `@renderer` → `src/renderer/src`. Talks to main via the preload API; talks to Convex directly. Entry `main.tsx`, HTML `src/renderer/index.html`.
- **Preload bridge** exposes `api.openExternal` / `onAuthCallback` / `getScreenSources` / `setScreenShareSource` / `setWindowFullScreen` / `onWindowFullScreenChange` (all `ipcRenderer.invoke`/`on`) + a **minimal `window.electron` marker**. **CRITICAL: the preload uses ONLY `electron`'s `contextBridge`+`ipcRenderer` — do NOT import `@electron-toolkit/preload`** (its `electronAPI` throws under `sandbox: true`, and a throwing preload exposes NOTHING → `window.api` undefined → every native call silently no-ops; this was a real multi-day bug behind broken screen share / fullscreen / auth-callback). `platform.hasNativeBridge` (`Boolean(window.api)`) is the honest "the bridge loaded" check, distinct from `isElectron` (which UA-detects Electron even if the bridge failed).

## Security / auth invariants (audited 2026-07-10)

- **Identity is keyed on `identity.tokenIdentifier`** (`${issuer}|${subject}`), never on `identity.subject` alone — the Convex guidelines forbid the latter as a global identity key. `users.tokenIdentifier` + `by_token_identifier`; `users.externalId` still holds the raw subject. `getCurrentUser` falls back to `by_external_id` for rows written before the field existed, and `users.store` backfills it on the next sign-in.
- **Invitations are capability tokens, not email trust (CLOSED — was the unverified-email hole).** `users.email` is still UNVERIFIED (WorkOS's access token carries no `email` claim, so `users.store` takes the address from the client on first sign-in; `store` refuses to _change_ an existing row's email, `emailVerified` records provenance) — so **authorization must never key on it.** Invitations now mint a **secret token** (`makeToken()`, ~120 bits, `workspaceInvitations.token` + `by_token` index): `invite` returns `{ code }` (email is an optional _note_, not a gate), `preview({code})` shows the join screen, `acceptByToken({code})` redeems it. The old email-trust surface (`myInvitations`/`accept`/`decline`) is **removed**. Joining requires the code (shared out-of-band), so a brand-new identity claiming any address gains nothing. Treat the token as the capability; don't reintroduce email-based auto-join. (`WORKOS_API_KEY` server-side email verification remains an option if you ever want verified addresses for other reasons, but invitations no longer need it.)
- **Every mutation derives the caller server-side.** No function takes a `userId` for authorization. Actions (`gifs.search`, `unsplash.search`/`trackDownload`) call `requireIdentity` — they spend our KLIPY/Unsplash quota with server-side keys.
- **Kind gates**: `messages.send` requires `channel.kind === 'chat'` and `pages.saveContent`/`saveMeta` require `kind === 'page'`, so a crafted call can't bury rows in a channel no view surfaces. `assertGroupInWorkspace` stops `channels.create`/`move`/`reorder` pointing a channel at another workspace's group (nothing else would ever clean that up).
- **Cascade deletes are batched (was the `.collect()` debt — DONE).** `channels.remove` / `workspaces.remove` / `threads.remove` no longer `.collect()` their cascades (which would exceed a mutation's document limit at scale and fail _every_ time). They delete the top row now + hand the cascade to **`convex/cleanup.ts`** internal mutations (`channel`/`thread`/`workspace`/`member`, `BATCH=50`): each `.take(BATCH)`, purges the batch (a message's reactions + R2 attachments too), and `ctx.scheduler.runAfter(0, internal.cleanup.X, …)` re-schedules itself while a batch stays full. `workspace` fans out a `cleanup.channel` per channel, then drains groups/members/invitations. `channel` drains messages (incl. thread replies, which carry `channelId`), notifications, thread rows, channelReads, kanban tasks/columns + the page (+ R2 cover). **`member`** drains a departed member's channelReads + inbox notifications for that workspace (scheduled by `members.remove` / `workspaces.leave`; their authored messages stay, Slack/Discord-style). So a delete returns immediately and completes in the background regardless of size. **Cascade coverage** (every child table has a delete path): reactions←message, attachments(R2)←message, notifications←message/channel/member, channelReads←channel/member, threads+replies←channel/message-root, pages/kanban←channel, groups/members/invitations+logo(R2)←workspace.

## Engineering rules

_(Adopted from `zinx-os`'s CLAUDE.md, adapted to this stack — pnpm + electron-vite + TanStack Router + Zustand.)_

### TypeScript

- **Never use `any`.** Use `unknown` + a narrow, or derive a precise type. (The only `any`s in the repo are in generated `routeTree.gen.ts` — never hand-edit it.)
- **Infer types from Convex functions** instead of redeclaring them:
  ```ts
  import type { FunctionReturnType, FunctionArgs } from 'convex/server'
  import { api } from '@convex/_generated/api'

  type ChannelMessage = FunctionReturnType<typeof api.messages.listByChannel>[number]
  type SendArgs = FunctionArgs<typeof api.messages.send>
  ```
- Use `Doc<'table'>` / `Id<'table'>` from `@convex/_generated/dataModel` when handling documents.
- Run `pnpm typecheck && pnpm lint` after **every** task. Never call work complete with type/lint errors you introduced.

### Convex queries

- **Always `.withIndex(...)` / `.withSearchIndex(...)`. Never `.filter(...)`** on a query builder — if the index doesn't exist, add it to `convex/schema.ts`. (JS `array.filter()` on an already-fetched result is fine and unrelated.)
- **Never `.collect()` unbounded data.** Use `.paginate(...)`, or `.take(n)` when you genuinely want a fixed small N (e.g. `messages.listByChannel` takes the newest 200). `.collect()` is only OK when the result is provably small and bounded (a workspace's channels/groups/members, one message's reactions).
- Known debt: `workspaces.remove` / `channels.remove` `.collect()` their cascades — fine at current scale, should become paginated/batched deletes.

### Error handling

- **Throw `ConvexError`, never a plain `Error`**, from Convex functions. Convex only ships `ConvexError.data` to the client; a plain `Error` surfaces as an opaque `"Server Error"` + request id.
  ```ts
  import { ConvexError } from 'convex/values'
  throw new ConvexError('Only owners and admins can change roles')
  ```
- **On the client**, read it back with `lib/convex-error.ts` `errorMessage(err, fallback)` — it returns the `ConvexError` message and replaces anything unrecognised with `fallback`.
- **Never expose internals**: stack traces, raw error objects, request ids, env values, or upstream service messages must not reach the UI. Show a friendly message; log the full error to the console.
- Loading and empty states are part of error handling — handle `undefined` (loading) and `null` (missing) query results explicitly. Our queries are deliberately **null-safe** (return `[]`/`null` rather than throwing) so a first-login race can't blow up the UI.

### Convex components

If a published Convex component already solves a problem (auth, files, rate limiting, RAG, agents, crons…), **use it instead of rolling your own** — https://www.convex.dev/components. **Always ask before adding one**: they bring deps, schema, env vars, and are non-trivial to remove. **In use:** `@convex-dev/r2` (file uploads — see the file-uploads bullet) · `@convex-dev/resend` (transactional email / invites — `convex/resend.ts` + `convex/email.ts`, `RESEND_API_KEY` env; see the shared-channels + invitations bullets). Components are registered in `convex/convex.config.ts`.

### CSS: viewport units

- **Always `dvh`/`dvw`, never `vh`/`vw`/`svh`/`lvh`** in app code (`h-dvh`, `max-h-[90dvh]`, `w-[92dvw]`). `dvh` tracks the mobile URL bar — this matters because the renderer also ships as a **web app**.
- Exception: `components/ui/**` is shadcn-vendored — don't edit it.

### UI: never expose internal ids to users

- In selects, dropdowns, command menus, autocompletes, toasts, tooltips and error text: **the visible label must be a human-readable name** (and optionally email) — never an `Id<'…'>`, `externalId`, or slug-token. The id stays the `value` carried by state; only the _displayed text_ must be readable.
- Sentinel values must show a human label (`"Ungrouped"`, not `__ungrouped__`).
- Exception: an explicit **"Copy channel ID" / "Copy user ID"** action (Discord-style developer affordance) — it _copies_ an id, it never _displays_ one as a label.
- For entities without a display name on the row, cache it at write-time rather than doing read-time joins on every render.

### Testing

**Harness = `vitest` + `convex-test` (`pnpm test`).** `vitest.config.ts` runs two projects: **`convex`** (`environment: 'edge-runtime'`, `@edge-runtime/vm`, `convex/**/*.test.ts`) and **`renderer`** (`environment: 'node'`, `@renderer`/`@convex` aliases, `src/renderer/**/*.test.ts`). Don't mock Convex — `convex-test` is the real in-memory harness (`convexTest(schema, import.meta.glob('./**/*.ts'))`; drive auth with `t.withIdentity({ subject, issuer, tokenIdentifier })`). A `.test.ts` file in `convex/` is **not** deployed (Convex excludes them) and is eslint-ignored (`convex/**`); renderer test files get an eslint override that relaxes the return-type rule.

- **`convex/domain.test.ts`** covers the security- + cascade-critical backend: every cascade delete (channel → messages/reactions/threads/replies, **thread-root message → whole thread**, workspace → channels/members/messages), invite-token accept/reject, non-member-post rejection, home-channel delete guard. **Cascade tests need fake timers** (`vi.useFakeTimers()`) + `await t.finishAllScheduledFunctions(vi.runAllTimers)` to drive the self-rescheduling `runAfter(0)` cleanup chain — `finishInProgressScheduledFunctions()` alone only awaits already-running jobs, not pending ones, so it silently no-ops on a fresh queue.
- **Renderer specs** (`lib/*.test.ts`) cover the load-bearing pure logic: `buildMessageRows` (the 8-message grouping cap + author/gap/reply breaks + day dividers), `messagePreview`/`gifSrc`, and the `zinx://` mention link round-trip + `stripMentionLinks`. Keep renderer tests to pure modules (no DOM) so the node project needs no jsdom.
- Run `pnpm test` in the workflow checklist after touching backend cascade logic or any of those pure modules.

### Workflow checklist

1. Make the smallest change that solves the task.
2. `pnpm typecheck` — must pass.
3. `pnpm lint` — must pass.
4. Touched Convex? `npx convex dev --once` to push schema/functions.
5. Touched backend cascade logic or a covered pure module (`message-rows`/`message-preview`/`mention`)? `pnpm test` — must pass.
6. Touched UI? Say what to verify in the app (you can't see it).
7. Don't commit unless explicitly asked.

## Non-negotiables

- **Security:** keep `contextIsolation: true`, `nodeIntegration: false`, `sandbox: true` (set explicitly in `src/main/index.ts` — the electron-vite template ships `sandbox: false`; **we override it**, don't revert). Validate every IPC input in main. Never put secrets/API keys in the renderer — proxy through Convex or main.
- Main ↔ renderer only via `contextBridge` + `ipcMain.handle` / `ipcRenderer.invoke`. Keep `setWindowOpenHandler` denying new windows.
- Don't add `openDevTools()` to shipped code (F12 in dev is fine — handled by `@electron-toolkit/utils` `optimizer`).

## pnpm / packaging notes

- **Build-script allow-list lives in `pnpm-workspace.yaml`** (`allowBuilds:` — electron, esbuild, electron-winstaller). pnpm 11 no longer reads the `pnpm` field in `package.json`; if esbuild/electron native binaries fail to build, that's why.
