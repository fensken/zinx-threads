# zinx-threads

Slack-style team chat. **electron-vite (Vite + TypeScript)** desktop shell; **React 19** UI + **Convex** backend (**full data model live ‚Äî chat/threads/pages/kanban/voice/shared-channels all real; the app is Convex-only, the mock/demo scaffolding was removed 2026-07-12**, see memory `keep-mock-fallback`) + **WorkOS AuthKit** auth (wired ‚Äî Path A auto-provision). Electron 39. **pnpm, not npm.**

**Branding + app constants = ONE source of truth, `src/shared/brand.ts`** (`BRAND.productName`/`company`/`appId`/`primaryHex`/`domain`/`supportEmail` + `DEEP_LINK_SCHEME` `zinx` + `DESKTOP_REDIRECT_URI` `http://127.0.0.1:9876/callback` + `TITLE_BAR_HEIGHT` `44`). Zero-dep so it's safe in every process; imported by the **renderer** via the **`@shared`** alias (wired into `electron.vite.config.ts` + `vite.web.config.ts` + `tsconfig.web`/`tsconfig.node`) and by **main/preload** via relative `../shared/brand`. Three surfaces CAN'T import it (separate bundles / not TS) and **mirror it by hand** ‚Äî `convex/lib/brand.ts` (Convex bundles only `convex/`: product name, `primaryHex`, `API_TOKEN_PREFIX` `zt_`), `electron-builder.yml` (appId/productName/scheme), and `src/renderer/index.html` (`<title>`/description/theme-color); each carries a "mirrors brand.ts" comment. **Rename/re-skin = edit `brand.ts` + the two convex/yaml/html mirrors.** **Product name = "Zinx Threads"** (display everywhere: `<title>`, sign-in, onboarding, settings, electron `productName`/`appId` `com.zinxthreads.app`). **Brand logo = a real image, `resources/logo.png` (1024√ó1024, the master), fanned out to every surface by `scripts/generate-icon.mjs`** ‚Äî a **pure-Node PNG resizer** (decode/encode via the built-in `zlib`, box-average premultiplied downscale; NO native deps, works offline ‚Äî same "no native deps" reason the old "rose Z" generator hand-rolled its encoder). It emits each surface at a size that fits its job, rather than a 1024¬≤ copy everywhere (a 1024¬≤ PNG as a 16px favicon is ~1.3 MB down the wire for nothing). `Logo` (`components/layout/logo.tsx`) renders that image (`<img src={logoSrc}>` clipped `rounded-xl object-cover`) ‚Äî **imported as a module** (`@renderer/assets/logo.png`), NOT a `public/` path, so the URL resolves under the packaged `file://` renderer; `LogoWordmark` = mark + name. The generator writes: `build/icon.png` (**1024**, verbatim master ‚Äî electron-builder's source; it regenerates `icon.ico`/`icon.icns` for the Windows taskbar + macOS dock at package time) + `resources/icon.png` (**512** ‚Äî Linux window/runtime + the tray source, resized to 16px in `system-integration.ts`) + `src/renderer/src/assets/logo.png` (**256** ‚Äî the in-app import) + `src/renderer/public/logo.png` (**256** ‚Äî general web logo at `/logo.png`) + `src/renderer/public/apple-touch-icon.png` (**180**) + `src/renderer/public/favicon-32.png`/`favicon-16.png` (**32**/**16** ‚Äî the browser tab favicon, `<link rel=icon sizes=‚Ä¶>` + `<link rel=apple-touch-icon>` in `index.html`; the master requires an 8-bit RGB/RGBA non-interlaced PNG, which the decoder enforces with a clear error). **To rebrand: replace `resources/logo.png`, run `node scripts/generate-icon.mjs`, rebuild.** (The old `favicon.svg` is unused; `theme-color` stays `#e11d48`.)

**One codebase ‚Üí desktop AND web.** The `src/renderer` app is the product; the Electron main/preload are just the desktop shell. It also builds as a plain web app (`vite.web.config.ts`) because data goes straight to Convex (works anywhere), native calls go through `src/renderer/src/lib/platform.ts` (feature-detect + web fallback ‚Äî renderer code must NOT touch `window.electron`/`window.api` directly), and `router.ts` picks history per target (`isPackagedDesktop` ‚Üí hash for `file://`; web AND `pnpm dev` ‚Üí browser history). Keep the renderer web-portable: no Electron/Node imports in `src/renderer`, always go through the `platform` layer.

## Commands

- `pnpm dev` ‚Äî desktop app with HMR (renderer) + hot-reload (main/preload)
- `pnpm dev:web` ‚Äî the same renderer as a **web app** (Vite, http://localhost:5173)
- `pnpm build` ‚Äî typecheck, then `electron-vite build` ‚Üí `out/` (desktop)
- `pnpm build:web` ‚Äî typecheck:web, then `vite build` ‚Üí `out/web` (web) ¬∑ `pnpm preview:web`
- `pnpm start` ‚Äî preview a production desktop build (`electron-vite preview`)
- `pnpm lint` ‚Äî ESLint ¬∑ `pnpm typecheck` ‚Äî tsc (node + web) ¬∑ `pnpm format` ‚Äî Prettier ¬∑ `pnpm test` ‚Äî vitest (renderer logic + `convex-test` backend; `test:watch` to watch)
- `pnpm build:win` ‚Äî build + package installer via electron-builder (also `build:mac`, `build:linux`)

## Process model

Three isolated processes, one `electron.vite.config.ts` with a `main` / `preload` / `renderer` section each.

- `src/main/index.ts` ‚Äî main process (Node): windows, app lifecycle, OS APIs. Uses `@electron-toolkit/utils`.
- `src/preload/index.ts` ‚Äî bridge: expose a _narrow, typed_ API via `contextBridge.exposeInMainWorld`. No raw `ipcRenderer`, no Node, to the renderer. Types in `src/preload/index.d.ts` (augments `window.electron` / `window.api`).
- `src/renderer/src/` ‚Äî renderer (browser): React UI only. `@renderer` ‚Üí `src/renderer/src`. Talks to main via the preload API; talks to Convex directly. Entry `main.tsx`, HTML `src/renderer/index.html`.
- **Preload bridge** exposes `api.openExternal` / `onAuthCallback` / `getScreenSources` / `setScreenShareSource` / `setWindowFullScreen` / `onWindowFullScreenChange` (all `ipcRenderer.invoke`/`on`) + a **minimal `window.electron` marker**. **CRITICAL: the preload uses ONLY `electron`'s `contextBridge`+`ipcRenderer` ‚Äî do NOT import `@electron-toolkit/preload`** (its `electronAPI` throws under `sandbox: true`, and a throwing preload exposes NOTHING ‚Üí `window.api` undefined ‚Üí every native call silently no-ops; this was a real multi-day bug behind broken screen share / fullscreen / auth-callback). `platform.hasNativeBridge` (`Boolean(window.api)`) is the honest "the bridge loaded" check, distinct from `isElectron` (which UA-detects Electron even if the bridge failed).

- **Permissions = Slack's model, not Discord's (LIVE).** Two questions, kept apart because they have different answers: **who can SEE a channel** (`channels.visibility` = `public` | `private`) and **who can POST in it** (`channels.postingPolicy` = `everyone` | `admins` | `selected`). A read-only channel is fully visible; a private channel is fully writable by the people in it. Collapsing the two into one "permission level" is what makes a role matrix hard to reason about ‚Äî `_zinx` uses **rank thresholds** (admin > member, so an admin reads everything) and `zinx-os` a full RBAC+ABAC matrix; we took neither. **Content access is decided by MEMBERSHIP, not rank** ‚Äî because "these five people" is not a rank, and no ladder can express it.

- **Sound design = `lib/sounds.ts` (rewritten 2026-07-14) ‚Äî synthesized, no audio assets.** One shared `AudioContext` (the old code made a new one per sound, and browsers cap how many you can have ‚Äî a busy call would eventually go silent). **Calm, not intrusive; clear, not loud.** Three things separate a chime from a beep, and the old version broke all three: a **soft attack** (a tone that jumps to full volume in 1ms _clicks_ ‚Äî everything ramps over 15‚Äì30ms, which is what makes a note feel _struck_), a **long exponential release** (resonant things decay slowly and never quite stop; a fixed-length note sounds electronic), and **overtones + a lowpass** (a bare sine is thin ‚Äî a fundamental plus a quiet octave, rolled off, carries at a _lower_ volume than a naked sine needs). ## Security / auth invariants (audited 2026-07-10)

- **Identity is keyed on `identity.tokenIdentifier`** (`${issuer}|${subject}`), never on `identity.subject` alone ‚Äî the Convex guidelines forbid the latter as a global identity key. `users.tokenIdentifier` + `by_token_identifier`; `users.externalId` still holds the raw subject. `getCurrentUser` falls back to `by_external_id` for rows written before the field existed, and `users.store` backfills it on the next sign-in.
- **Invitations are capability tokens, not email trust (CLOSED ‚Äî was the unverified-email hole).** `users.email` is still UNVERIFIED (WorkOS's access token carries no `email` claim, so `users.store` takes the address from the client on first sign-in; `store` refuses to _change_ an existing row's email, `emailVerified` records provenance) ‚Äî so **authorization must never key on it.** Invitations now mint a **secret token** (`makeToken()`, ~120 bits, `workspaceInvitations.token` + `by_token` index): `invite` returns `{ code }` (email is an optional _note_, not a gate), `preview({code})` shows the join screen, `acceptByToken({code})` redeems it. The old email-trust surface (`myInvitations`/`accept`/`decline`) is **removed**. Joining requires the code (shared out-of-band), so a brand-new identity claiming any address gains nothing. Treat the token as the capability; don't reintroduce email-based auto-join. (`WORKOS_API_KEY` server-side email verification remains an option if you ever want verified addresses for other reasons, but invitations no longer need it.)
- **Every mutation derives the caller server-side.** No function takes a `userId` for authorization. Actions (`gifs.search`, `unsplash.search`/`trackDownload`) call `requireIdentity` ‚Äî they spend our KLIPY/Unsplash quota with server-side keys.
- **Kind gates**: `messages.send` requires `channel.kind === 'chat'` and `pages.saveContent`/`saveMeta` require `kind === 'page'`, so a crafted call can't bury rows in a channel no view surfaces. `assertGroupInWorkspace` stops `channels.create`/`move`/`reorder` pointing a channel at another workspace's group (nothing else would ever clean that up).
- **Cascade deletes are batched (was the `.collect()` debt ‚Äî DONE).** `channels.remove` / `workspaces.remove` / `threads.remove` no longer `.collect()` their cascades (which would exceed a mutation's document limit at scale and fail _every_ time). They delete the top row now + hand the cascade to **`convex/cleanup.ts`** internal mutations (`channel`/`thread`/`workspace`/`member`, `BATCH=50`): each `.take(BATCH)`, purges the batch (a message's reactions + R2 attachments too), and `ctx.scheduler.runAfter(0, internal.cleanup.X, ‚Ä¶)` re-schedules itself while a batch stays full. `workspace` fans out a `cleanup.channel` per channel, then drains groups/members/invitations. `channel` drains messages (incl. thread replies, which carry `channelId`), notifications, thread rows, channelReads, kanban tasks/columns + the page (+ R2 cover). **`member`** drains a departed member's channelReads + inbox notifications for that workspace (scheduled by `members.remove` / `workspaces.leave`; their authored messages stay, Slack/Discord-style). So a delete returns immediately and completes in the background regardless of size. **Cascade coverage** (every child table has a delete path): reactions‚Üêmessage, attachments(R2)‚Üêmessage, notifications‚Üêmessage/channel/member, channelReads‚Üêchannel/member, threads+replies‚Üêchannel/message-root, pages/kanban‚Üêchannel, groups/members/invitations+logo(R2)‚Üêworkspace.

## Engineering rules

_(Adopted from `zinx-os`'s CLAUDE.md, adapted to this stack ‚Äî pnpm + electron-vite + TanStack Router + Zustand.)_

### The ripple rule ‚Äî update every place a feature is used

- **A feature is never edited in isolation. When you add, change, or extend one, find every place it is used or depended on and update those too ‚Äî in the SAME change.** A capability in this app is deliberately consumed through several surfaces, and leaving one behind is how they silently drift out of sync:
  - **The developer platform is ONE catalog, three surfaces.** Adding or changing a capability means updating `convex/apiTools.ts` (the impl) **and** the `TOOLS`/`callTool` dispatch in `convex/lib/mcp.ts` **and** the `/docs` capability table ‚Äî so it lands in **MCP, the REST API, and bots at once**. Never wire a feature into one surface only (see the developer-platform section + memory `mcp-connector`).
  - **A `bot` is a real user principal.** Anything on the message/notification/permission path (a new gate, a new `MessageAuthor` field, a new fan-out branch) must keep working when the actor is a bot, and its avatar/badge must resolve (`avatarImageFor`, `AuthorRoleBadge`).
  - **Offline/local parity is mandatory** (memory `offline-parity-rule`): any feature that can run without a server must ALSO exist in **local mode** (`/local`), looking the same minus the online-only parts ‚Äî and the reverse: a new local channel kind or data shape must be taught to `src/main/local-data.ts` (`REL_PATH_RE`/`SUB_DIRS`), `lib/local-data.ts` (`parseSnapshot`/`buildFileMap`), the local store, AND the export/import format, or it is silently dropped.
  - **The load-bearing enumerations** ‚Äî anything that lists channels, threads, members, unread, notifications, search ‚Äî is where a new channel kind, permission, or DM/shared-channel leaks (see the permissions + DM + shared-channel sections). Teach every enumeration about the new thing, not just the happy-path reader.
  - **Shared UI + serializers**: the composer, `MarkdownMessage`, `docToMarkdown`/`markdownToHtml`, `messagePreview`, the mention scheme, `DateTimePicker`, the skeletons ‚Äî each has multiple call sites. Change the contract in one, audit the others.
  - **The test**: after a change, ask "what reads this table / calls this function / renders this shape / consumes this capability?" and grep for it. If a dependent isn't updated, say so explicitly rather than leaving it stale.

### The comment-as-smell rule

- **If you need a paragraph-long comment to justify why a workaround is OK, the code is wrong ‚Äî fix the code, don't defend it.** A long comment explaining _why_ a genuinely non-obvious approach is correct (a browser quirk, a spec constraint, a race we must win) is fine and welcome ‚Äî this codebase is deliberately verbose about _why_. But a paragraph arguing that a hack, a ref-juggle, a hand-rolled version of something we already have, or a "don't touch this or it breaks" is _acceptable_ is a signal to restructure until the justification is unnecessary. The test: could the same guarantee come from an existing helper, a different shape, or a named abstraction? Then reach for that. Example (2026-07-15): `whiteboard-view.tsx` hand-rolled a debounce with `dirty`/`timer`/`persistRef` and two paragraphs defending the ref juggling and the unmount flush ‚Äî all of which `lib/use-debounced-callback.ts` already does; the fix was to delete the machinery and call the hook, and the paragraphs went with it.

### TypeScript

- **Never use `any`.** Use `unknown` + a narrow, or derive a precise type. (The only `any`s in the repo are in generated `routeTree.gen.ts` ‚Äî never hand-edit it.)
- **Infer types from Convex functions** instead of redeclaring them:
  ```ts
  import type { FunctionReturnType, FunctionArgs } from 'convex/server'
  import { api } from '@convex/_generated/api'

  type ChannelMessage = FunctionReturnType<typeof api.messages.listByChannel>[number]
  type SendArgs = FunctionArgs<typeof api.messages.send>
  ```
- Use `Doc<'table'>` / `Id<'table'>` from `@convex/_generated/dataModel` when handling documents.
- Run `pnpm typecheck && pnpm lint` after **every** task. Never call work complete with type/lint errors you introduced.

### Convex queries

- **Always `.withIndex(...)` / `.withSearchIndex(...)`. Never `.filter(...)`** on a query builder ‚Äî if the index doesn't exist, add it to `convex/schema.ts`. (JS `array.filter()` on an already-fetched result is fine and unrelated.)
- **Never `.collect()` unbounded data.** Use `.paginate(...)`, or `.take(n)` when you genuinely want a fixed small N (e.g. `messages.listByChannel` takes the newest 200). `.collect()` is only OK when the result is provably small and bounded (a workspace's channels/groups/members, one message's reactions).
- **Never filter in JS what an index can exclude.** `.collect()` + `array.filter()` still _reads_ every row it throws away ‚Äî and the rows you're dropping are usually the ones that grow. The canonical case: **`channels` holds DMs too**, and DMs grow with the square of the member count, so `by_workspace` + `.filter(c => c.kind !== 'dm')` read thousands of private rows on queries that run for every member on every message. Use **`lib/channels.ts` `listRealChannels`** (one range per real kind, via `by_workspace_kind`) ‚Äî never re-hand-roll the scan.
- **Watch what a mutation _invalidates_, not just what it reads.** Convex re-runs every query that read a document you patch, on every subscribed client. So a field that changes constantly must not sit on a document that quiet queries read. `channels.lastMessageAt` did, and one message re-ran the sidebar, the route resolver, the workspace directory, the ‚åòK palette, the threads flyout and the DM list ‚Äî for everyone. It now lives in **`channelActivity`** (`lib/activity.ts` `bumpChannelActivity` / `watermarks`), which only unread + the DM list read. Same rule behind: never patch a document with the value it already has (`markChannelRead`, `pages.saveContent`, `bumpChannelActivity` all guard) ‚Äî a no-op write still re-notifies every subscriber.
- **Denormalise what is read constantly and written rarely.** `messages.reactions` is a summary (`{emoji, count, userIds}`, `lib/reactions.ts`) kept on the message doc by `toggleReaction`; `messageReactions` stays the source of truth (it's what makes one-per-user-per-emoji enforceable). Rendering a page used to cost **one index range per message** on the hottest query in the app. Reads fall back to the table when the summary is absent (pre-existing rows), and `migrations.backfillReactions` fills them in ‚Äî new messages are born with `reactions: []`.
- **An R2 key is not a secret.** Every mutation that adopts an uploaded `key` goes through `files.ts` `markUploadUsed(ctx, userId, key)`, which requires the key to be an **unattached upload owned by the caller**. Keys are visible in attachment URLs, so without that check you could attach someone else's file to your own message and then delete the message ‚Äî `messages.remove` deletes its attachments' R2 objects, taking their file with it.
- Cascade deletes are already batched + self-rescheduling (`convex/cleanup.ts`); a new child table needs a delete path there, or it outlives its parent.

### Error handling

- **Throw `ConvexError`, never a plain `Error`**, from Convex functions. Convex only ships `ConvexError.data` to the client; a plain `Error` surfaces as an opaque `"Server Error"` + request id.
  ```ts
  import { ConvexError } from 'convex/values'
  throw new ConvexError('Only owners and admins can change roles')
  ```
- **On the client**, read it back with `lib/convex-error.ts` `errorMessage(err, fallback)` ‚Äî it returns the `ConvexError` message and replaces anything unrecognised with `fallback`.
- **Never expose internals**: stack traces, raw error objects, request ids, env values, or upstream service messages must not reach the UI. Show a friendly message; log the full error to the console.
- Loading and empty states are part of error handling ‚Äî handle `undefined` (loading) and `null` (missing) query results explicitly. Our queries are deliberately **null-safe** (return `[]`/`null` rather than throwing) so a first-login race can't blow up the UI.

### Convex components

If a published Convex component already solves a problem (auth, files, rate limiting, RAG, agents, crons‚Ä¶), **use it instead of rolling your own** ‚Äî https://www.convex.dev/components. **Always ask before adding one**: they bring deps, schema, env vars, and are non-trivial to remove. **In use:** `@convex-dev/r2` (file uploads ‚Äî see the file-uploads bullet) ¬∑ `@convex-dev/resend` (transactional email / invites ‚Äî `convex/resend.ts` + `convex/email.ts`, `RESEND_API_KEY` env; see the shared-channels + invitations bullets). Components are registered in `convex/convex.config.ts`.

### CSS: colour

- **Never hardcode a colour.** Semantic tokens only (`bg-card`, `text-muted-foreground`, `border-border`). The palette is swapped by re-running the tweakcn installer, and a literal survives the swap and clashes with the new one. Two documented exceptions, both _categorical_ rather than brand: **avatar swatches** (`account-controls.tsx` ‚Äî user data, and they must stay distinguishable from each other) and **status colours** (presence dots, the `--speaking` ring, the amber mention highlight ‚Äî they mean "away"/"talking"/"this pings you", so a theme with a green or amber primary must not make them read as an ordinary accent). Scrims and `text-white` **over an image** are also fine ‚Äî they're relative to the photo, not the theme.
- **Never dim TEXT with an opacity modifier.** `text-foreground/70` is not a colour, it's a dimmer ‚Äî and on a low-contrast palette it takes text straight past legibility (it did: the sidebar was unreadable in light mode). The theme ships exactly the two you need: **`foreground`** for primary text and **`muted-foreground`** for secondary. Use them. Opacity on a whole _element_ (a disabled button, a pending message row) is fine; opacity on a text colour is not.
- **Buttons are shadcn `Button`** ‚Äî `variant`/`size`, not a hand-rolled `<button>` with its own Tailwind. A bespoke button drifts from the rest of the app on hover, focus ring, disabled state and dark mode, and has to be re-fixed by hand every time the palette moves. `common/icon-button.tsx` `IconButton` (a `ghost`+`icon` Button + tooltip) is the icon-only case and covers most of the app. Hand-roll only where `Button` genuinely can't go ‚Äî window chrome (`window-controls.tsx`, which needs exact OS metrics), and elements that are a _surface_ rather than a control (a channel row, a message row).
- **Contrast is measured, not eyeballed.** Every foreground/surface pair the app renders must clear **WCAG AA** (4.5:1 for body text). The tweakcn themes do NOT guarantee this ‚Äî the current one shipped a `--muted-foreground` that failed on `--muted` (light) and on `--accent` (dark), which is the hover surface of every icon button. It's pinned to a contrast floor in `globals.css`; **re-measure after any theme install**, since a new palette moves the surfaces too.
- **Use the token family that matches the SURFACE.** The sidebar sits on `--sidebar`, so its text is `sidebar-foreground` ‚Äî not `foreground`, which is calibrated against `--background` and may be a completely different colour. Same for `--card`, `--popover`, `--titlebar`.

### CSS: viewport units

- **Always `dvh`/`dvw`, never `vh`/`vw`/`svh`/`lvh`** in app code (`h-dvh`, `max-h-[90dvh]`, `w-[92dvw]`). `dvh` tracks the mobile URL bar ‚Äî this matters because the renderer also ships as a **web app**.
- Exception: `components/ui/**` is shadcn-vendored ‚Äî don't edit it.

### UI: never expose internal ids to users ‚Äî a HARD rule

_(Adopted verbatim in spirit from `zinx-os`'s CLAUDE.md, where it is also a hard rule. It has been violated twice here, both times in a `Select`; see the Base UI trap below.)_

- In selects, dropdowns, command menus, autocompletes, toasts, tooltips and error text: **the visible label must be a human-readable name** (and optionally email) ‚Äî never an `Id<'‚Ä¶'>`, `externalId`, `tokenIdentifier`, or slug-token. The id stays the `value` carried by state; **only the _displayed text_ is ever a name.**
- **Sentinel values must show a human label** ‚Äî `"No channel"` / `"Ungrouped"` / `"Anyone"`, never the sentinel string. **`__none__` must never reach the screen.**
- **The Base UI trap (this is the actual bug, twice).** `Select.Value` has no way to know an item's label, so **it prints the raw `value` as a fallback** ‚Äî a Select whose values are ids renders an id the moment one is picked, and one whose values are `'7'`/`'member'` renders `7` and `member`. **Always pass `items` to `<Select>`** (a `Record<value, label>`), which is what makes `Select.Value` render the label. Build the items map once and drive `SelectItem` from the same map, so the label can't drift from the trigger.
- Exception: an explicit **"Copy channel ID" / "Copy user ID"** action (Discord-style developer affordance) ‚Äî it _copies_ an id, it never _displays_ one as a label.
- The same applies to derived names: a **DM's** stored `name` is `dm-<participant ids>` ‚Äî an id in disguise ‚Äî so every surface (sidebar, inbox, search, event pickers) derives its title from the _people_ (`lib/dm.ts` `dmTitle`) and never renders `channel.name`.
- For entities without a display name on the row, cache it at write-time rather than doing read-time joins on every render.

### Testing

**Harness = `vitest` + `convex-test` (`pnpm test`).** `vitest.config.ts` runs two projects: **`convex`** (`environment: 'edge-runtime'`, `@edge-runtime/vm`, `convex/**/*.test.ts`) and **`renderer`** (`environment: 'node'`, `@renderer`/`@convex` aliases, `src/renderer/**/*.test.ts`). Don't mock Convex ‚Äî `convex-test` is the real in-memory harness (`convexTest(schema, import.meta.glob('./**/*.ts'))`; drive auth with `t.withIdentity({ subject, issuer, tokenIdentifier })`). A `.test.ts` file in `convex/` is **not** deployed (Convex excludes them) and is eslint-ignored (`convex/**`); renderer test files get an eslint override that relaxes the return-type rule.

- **`convex/domain.test.ts`** covers the security- + cascade-critical backend: every cascade delete (channel ‚Üí messages/reactions/threads/replies, **thread-root message ‚Üí whole thread**, workspace ‚Üí channels/members/messages), invite-token accept/reject, non-member-post rejection, home-channel delete guard, and the **event/timezone rules** (an instant is stored, not a wall-clock; organiser-only edit; RSVP upsert; non-member sees nothing) and the **DM privacy gate** (a workspace member can't read/post in / see / list someone else's DM; `dms.open` is find-or-create from either side; DMs stay out of `channels.listBySlug`; no messaging a non-member; every DM message notifies + counts unread with no `@`; no threads in a DM; rename/move/delete refused). **Cascade tests need fake timers** (`vi.useFakeTimers()`) + `await t.finishAllScheduledFunctions(vi.runAllTimers)` to drive the self-rescheduling `runAfter(0)` cleanup chain ‚Äî `finishInProgressScheduledFunctions()` alone only awaits already-running jobs, not pending ones, so it silently no-ops on a fresh queue.
- **Renderer specs** (`lib/*.test.ts`) cover the load-bearing pure logic: `timezone`/`calendar-grid` (wall-clock‚ÜîUTC across zones AND either side of a DST switch ‚Äî the one conversion that fails silently), `buildMessageRows` (the 8-message grouping cap + author/gap/reply breaks + day dividers), `messagePreview`/`gifSrc`, and the `zinx://` mention link round-trip + `stripMentionLinks`. Keep renderer tests to pure modules (no DOM) so the node project needs no jsdom.
- Run `pnpm test` in the workflow checklist after touching backend cascade logic or any of those pure modules.

## pnpm / packaging notes

- **Build-script allow-list lives in `pnpm-workspace.yaml`** (`allowBuilds:` ‚Äî electron, esbuild, electron-winstaller). pnpm 11 no longer reads the `pnpm` field in `package.json`; if esbuild/electron native binaries fail to build, that's why.

- **Windows gotcha on this machine:** `pnpm add` / `pnpm remove` fail with `EPERM: ‚Ä¶ rename ‚Ä¶ package.json` (something locks the file). Workaround: hand-edit `package.json`, then run `pnpm install` (which doesn't rewrite it) to sync `node_modules` + lockfile.
- **Distribution + auto-update = GitHub Releases (LIVE, beta/unsigned ‚Äî see `RELEASING.md`).** Packaging config `electron-builder.yml`; `publish` targets `github` (`fensken/zinx-threads`). **`src/main/updater.ts` `initAutoUpdater(() => mainWindow)`** (called once from `whenReady`, packaged-only) checks Releases on launch + every 6h via `electron-updater`, downloads in the background, installs on quit ‚Äî wrapped so a failed/unsupported check only logs. **In-app "Update available" badge:** it broadcasts an `UpdateState` (`available`/`downloaded`/`version`/`url`) to the renderer (IPC `update:state` + `update:get-state`; `update:install` ‚Üí `quitAndInstall`), surfaced by **`layout/update-badge.tsx` `UpdateBadge`** ‚Äî a pill in the **title bar, just left of the window action buttons** (`title-bar.tsx`). `downloaded` (Win/Linux, staged) ‚Üí "Restart to update" installs immediately; `available` only (unsigned macOS, or pre-download) ‚Üí "Update available" opens the release page. Because electron-updater can't run on unsigned macOS, a **manual GitHub-releases check** (`fetch` the repo's releases, `isNewer` vs `app.getVersion()`) is the cross-platform "available" signal so mac still gets the badge. `platform.updates` is the web-safe wrapper (badge never shows on web). **Cut a release:** bump `package.json` version ‚Üí CI (`.github/workflows/release.yml`, tag `v*` ‚Üí win/mac/linux matrix ‚Üí **draft** Release) **or** local `pnpm release:win` (= `build && electron-builder --win --publish always`, needs `GH_TOKEN`). `build:win`/`build:mac`/`build:linux` still package **without** publishing (a hand-out installer, no auto-update). **Beta reality:** builds are **unsigned** ‚Üí auto-update works on **Windows + Linux**; **macOS** (Squirrel) refuses unsigned updates + Gatekeeper blocks first open (right-click ‚Üí Open) until an Apple Developer ID + `mac.notarize: true`. Windows shows a SmartScreen "unknown publisher" prompt until signed. Beta runs against the **dev** Convex backend ‚Äî the `VITE_*` client config baked at build time (dev deployment values from `.env.local`; in CI, repo **Variables** since they're public client config, not secrets). `dev-app-update.yml` mirrors the github provider for testing.
